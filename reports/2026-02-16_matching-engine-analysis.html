2. Individual vs. Organization Classification

Current Logic

Classification: is_individual_name()

Three-step heuristic applied to the Google Places name field:

Step 1: Organization keyword check
  If name contains: center, clinic, hospital, medical group, associates,
  institute, health, university, ucla, usc, cedars, providence, kaiser,
  department, practice, office of, foundation, network
  → Classify as ORGANIZATION

Step 2: Credential suffix check  
  If name matches: MD, DO, DDS, DMD, DPM, OD, PhD, NP, PA, APRN, FNP, DC, DPT
  → Classify as INDIVIDUAL

Step 3: Name structure check
  If name has 2-4 words, all capitalized
  → Classify as INDIVIDUAL

Default → ORGANIZATION (conservative)

3. NPI Matching Logic (Current)

Cascading Match Strategy

Strategy 1: Exact — Name + Zip 95% confidence

WHERE LOWER(last_org_name) = LOWER(:last)
  AND LOWER(first_name) LIKE LOWER(:first) || '%'
  AND zip5 = :zip
  AND entity_type_code = 'I'

Strongest signal. Zip5 narrows geography tightly. First name uses prefix match to handle middle initials in CMS ("Dung Gloria" matches "Dung").

Strategy 2: Fuzzy — Name + City + State 80% confidence

WHERE LOWER(last_org_name) = LOWER(:last)
  AND LOWER(first_name) LIKE LOWER(:first) || '%'
  AND LOWER(city) = LOWER(:city)
  AND state = :state

Fallback when zip parsing fails or provider has a different zip in CMS vs Google (common with suite vs building addresses).

Strategy 3: Loose — Last Name + State 50-65% confidence

WHERE LOWER(last_org_name) = LOWER(:last)
  AND state = :state
  [AND LOWER(city) = LOWER(:city)]  -- if available

Last resort. Confidence boosted by +10% if first initial matches, +15% if specialty hint matches CMS provider_type. High false positive risk with common names.

Name Parsing: parse_provider_name()

Input: "Sarah R. Rettinger, MD"
  → Strip credentials: "Sarah R. Rettinger"
  → Strip specialty suffixes: "Sarah R. Rettinger"  
  → Split: ["Sarah", "R.", "Rettinger"]
  → Result: {first: "Sarah", last: "Rettinger"}  ✅

Input: "Dr. Eiriny T. Eskander, MD"
  → Strip credentials: "Dr. Eiriny T. Eskander"
  → "Dr." NOT stripped!  ← BUG
  → Split: ["Dr.", "Eiriny", "T.", "Eskander"]
  → Result: {first: "Dr.", last: "Eskander"}  ❌

Input: "Nazemi Reza MD"
  → Strip credentials: "Nazemi Reza"
  → Split: ["Nazemi", "Reza"]
  → Result: {first: "Nazemi", last: "Reza"}
  → Actual: first=Reza, last=Nazemi  ❌ (reversed in Google)

Address Parsing: parse_address()

Input: "1831 Wilshire Blvd suite a, Santa Monica, CA 90403, USA"
  → zip5: "90403"  ✅
  → state: "CA"    ✅  
  → city: "Santa Monica"  ✅

Input: "100 UCLA, Medical Plaza #310, Los Angeles, CA 90095, USA"
  → zip5: "90095"  ✅
  → state: "CA"    ✅
  → city: "Medical Plaza #310"  ❌ (takes second-to-last comma segment)

5. Gap Analysis & Failure Modes

Why Matches Fail (Root Causes)

1. Name Parsing Failures (3 of 8 misses)

  • "Dr." prefix not stripped — treated as first name
  • Reversed name order — "Nazemi Reza" in Google vs "Reza Nazemi" in CMS
  • Middle initials — inconsistently present between sources
  • Hyphenated/compound last names — "Garcia-Lopez" vs "Garcia Lopez" vs "Lopez"

2. Address Mismatches (3 of 8 misses)

  • Different practice address — Doctor lists Beverly Hills office on Google, but CMS has their billing address in Redondo Beach
  • Multi-location providers — CMS has one address, Google shows another location
  • Institutional vs practice address — CMS address is "UCLA" zip, Google shows the specific clinic suite

3. Not in Medicare (2 of 8 misses)

  • Provider doesn't bill Medicare (younger doctors, concierge practices, Medicaid-only)
  • Provider is in NPPES (has NPI) but not in our Medicare claims tables
  • Fix: Also search raw_nppes and raw_dac_national as fallbacks

4. Organization Leakage (1+ missed opportunities)

  • "Dr. Friedman's Endocrinology Clinic" classified as org due to "Clinic" keyword, but contains a matchable doctor name
  • Organizations currently return zero intelligence — no attempt to find providers at that address

6. Proposed Improvements

6.1 Name Parsing v2

Quick Wins (double the match rate)

1. Strip "Dr." / "Dr " prefix before parsing
2. Try both name orders: (first last) AND (last first)
3. Strip middle initials: "Eiriny T." → "Eiriny"
4. Handle possessives: "Dr. Friedman's Clinic" → extract "Friedman"
5. Normalize unicode: accented characters, special chars
6. Handle compound names: try both "Garcia-Lopez" and "Garcia"

6.2 Multi-Source Address Matching

Address Tolerance Strategy

The core problem: CMS and Google often have different addresses for the same provider. Solution: match against ALL known addresses for a provider.

Data sources for addresses:
  1. core_providers         → Primary billing address
  2. practice_locations     → All group practice addresses (from reassignment data)
  3. raw_dac_national       → Care Compare listed addresses (multiple per NPI)
  4. raw_nppes              → NPPES registered address

Strategy:
  - Build address lookup: NPI → [all known zip5s, cities]
  - When matching: check if Google's zip5 appears in ANY of the provider's known locations
  - Bonus: use lat/lng proximity (within 5 miles) as a soft match signal

6.3 Organization Matching (New Capability)

Strategy: Reverse-lookup providers at an address

When we identify a result as an organization (clinic, center, hospital), instead of giving up, find ALL providers who practice at that location:

Step 1: Parse org address → zip5
Step 2: Query CMS for providers at that zip5 with matching specialty
Step 3: Cross-reference with Google Places name for embedded doctor names
Step 4: Return provider roster for that location

Example: "UCLA Endocrine Center" at 90095
  → Find all endocrinology providers with zip5=90095
  → Cross-reference with raw_dac_national group affiliations
  → Return: "12 endocrinologists practice here"
  → Show top providers by volume/quality

User value: "I'm looking at UCLA Endocrine Center — who works there and who should I talk to?"

6.4 Phone Number Matching

High-confidence signal, requires Place Details API

Cost: $0.017 per Place Details call ($3.40 for 200 providers/day)
Source: Google Places Details API → phone number
Match: NPPES has practice phone for 90%+ of providers

Strategy:
  - Only call Place Details for UNMATCHED results (reduce API spend)
  - Normalize phone: strip +1, parens, dashes → 10-digit
  - Match against raw_nppes "Provider Business Practice Location Address Telephone Number"
  - Phone match = 90% confidence (phones rarely shared between unrelated providers)

6.5 Web Crawl Enrichment

The "Secret Sauce" Layer

For unmatched results OR organizations, crawl the provider's website to extract structured data:

Trigger: Result has no CMS match AND has a website (from Place Details)
  
Crawl targets:
  1. Provider's own website (from Google Places)
  2. Hospital/practice "Find a Doctor" pages
  3. Healthgrades/Vitals/WebMD profiles
  4. NPI lookup services (npidb.org, nppes.cms.hhs.gov)

Extract via LLM:
  - Provider NPI (often listed on practice websites)
  - Full provider roster (from "Our Team" pages)
  - Specialties, board certifications
  - Insurance accepted
  - Education, fellowship details
  - Languages spoken

Example flow:
  "Dr. Friedman's Endocrinology Clinic" → no individual match
  → Crawl website → find NPI on "About" page
  → Match NPI to CMS → full intelligence
  → Also find 3 other providers on "Our Team" page
  → Match those too → return enriched org profile

Why this is the moat: Anyone can match names to NPI. Crawling websites, extracting provider rosters from "Our Team" pages, and linking them to CMS claims data at scale — that's genuinely hard and creates defensible value.

9. The Secret Sauce: What Makes This Defensible

What anyone can build

  • Google Places search wrapper
  • Basic NPI lookup by name
  • Single-source data display
  • Static provider directories

What we're building

  • Multi-strategy matching cascade
  • Cross-source intelligence synthesis
  • Organization → provider roster resolution
  • Web crawl NPI extraction at scale
  • Pre-computed match tables (cache)
  • Composite targeting scores

The Moat Stack

Layer 1: Data Assembly (already built)

90M+ rows across 30 tables from 6 public sources, ingested into a fast analytical DB. Not hard individually, but the combination is the foundation nobody else has pre-built for BD teams.

Layer 2: Entity Resolution (building now)

Linking Google Places listings to NPI to CMS claims to Open Payments to MIPS scores. The matching engine IS the product. Every improvement to match rate directly increases user value. This is where engineering effort compounds.

Layer 3: Intelligence Synthesis (next)

Raw data → actionable insight. A targeting score that combines volume, quality, industry relationships, and accessibility into a single "should I call this doctor?" signal. LLM-generated provider summaries. Competitive landscape analysis.

Layer 4: Network Effects (future)

As users search and validate matches, build a confirmed match table (Google Place ID → NPI). Every user's searches improve accuracy for everyone. This is impossible to replicate without the user base.
